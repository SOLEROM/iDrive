# Kids Rides & Classes — PWA

A **Progressive Web App** that helps parent groups coordinate children's
classes and shared rides. Installs on Android and iPhone from the browser —
no app store, no backend server owned by us.

**Stack:** React 18 · TypeScript · Vite · Vitest · Dexie (IndexedDB) ·
Workbox service worker · React Router.

---

## What it does

Parenting groups juggle a lot of logistics — who drives whom, which week a
child has music vs. football, who already committed to the Tuesday pickup.
This app makes that shared knowledge explicit and keeps it synced across
every parent's phone.

- **Private, per-parent storage** — each parent's children and private events
  live in their *own* Google Drive. No central server sees it.
- **Shared group coordination** — each parent group uses a shared Google
  Sheet for ride-relevant activities: who's going, who volunteered to drive,
  what's been completed.
- **Ride volunteering** — any parent can claim a ride leg (**TO**, **FROM**,
  or **BOTH**). The Rides Board shows what's unassigned, claimed, confirmed,
  or done.
- **Offline-first** — everything is cached locally in IndexedDB. Sync
  happens in the background when connectivity returns.
- **Installable** — add to Home Screen on iOS; browser install prompt on
  Android.

---

## Who it's for

- A group of 3–20 parents coordinating one or more children's activities
- Families who'd rather not push their logistics into Google Calendar or a
  chat group
- Anyone who wants their family data to stay in their own Drive, not a
  vendor's server

## How it's built

```
src/
├── domain/         ← pure TS: enums, models, recurrence, ride state machine, conflicts, config
├── storage/        ← IndexedDB (Dexie) + repositories + sync queue
├── remote/         ← DriveAdapter / SheetsAdapter interfaces + mock/ implementations
├── services/       ← SyncEngine — drains the queue, reconciles with Drive/Sheets
├── state/          ← React context: current parent, config, sync state
├── components/     ← shared UI (Header, TabBar, ChildDot, RideStatusChip, SyncBanner)
├── screens/        ← one file per screen (Dashboard, Children, Events, Rides, …)
├── pwa/            ← service-worker registration
├── lib/            ← format, useLiveQuery
├── App.tsx         ← router
└── main.tsx        ← entry

public/
├── manifest.webmanifest  ← generated by vite-plugin-pwa from vite.config.ts
├── icons/                ← app icons
└── favicon.svg

tests/               ← Vitest mirrors src/ structure (domain, remote, config, services)
```

The app is a **frontend shell over a backend-driven intelligence system**.
Right now the "backend" is mocked in-memory — the adapter boundary is where
real Google Drive / Sheets wiring will plug in.

## Get started

→ **[`gettingStarted.md`](./gettingStarted.md)** — install, run, test, build,
deploy.

→ **[`install.sh`](./install.sh)** — one-shot setup on a fresh Ubuntu host.

Quick start if Node 20+ is already installed:

```bash
npm install
npm run dev          # http://localhost:5173
npm test             # 44 Vitest cases
npm run build        # dist/ — deploy to any static host
```

---

## Installability

**Android (Chrome / Edge)** — tap the "Install app" prompt or browser menu →
*Install app*. The app opens full-screen like a native install.

**iPhone / iPad (Safari)** — tap Share → *Add to Home Screen*. iOS installs
the manifest as a standalone app; launches without the Safari chrome.

---

## Privacy posture

No accounts on our server — there is no "our server". Data lives in the
parent's Google Drive (private data) or the group's shared Google Sheet
(coordination data). The service worker and IndexedDB are entirely local
to the user's device. Clearing browser data wipes the local cache.
